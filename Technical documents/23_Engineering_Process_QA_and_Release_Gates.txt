1. CORE ACCEPTANCE WORKFLOW
---------------------------
All PRs touching the following directories MUST include and verify the 
Cross-cutting Acceptance Checklist:
- supabase/functions/**
- supabase/migrations/**
- vscode-extension/**
- docs/mcp/**

The checklist is located in AGENTS.md (Section 7) and is automatically prepended
to the PR template in [.github/pull_request_template.md].

2. CROSS-CUTTING ACCEPTANCE CHECKLIST
-------------------------------------
- [ ] AGENTS.md updated (where to change what / invariants)
- [ ] Trust console updated (if metrics/schema changed)
- [ ] Lifecycle controls safe (no dangerous purges/deletions)
- [ ] Security review (RBAC, SSRF, no secret logs, output caps+redaction)
- [ ] CI / Manual QA (verified via scripts/mcp-smoke-test.ts or rag-eval)

3. QA ASSETS INDEX (docs/process/QA_INDEX.md)
--------------------------------------------
The index centralizes all repeatable test harnesses:

| Asset              | Purpose                                | Core Command                |
|--------------------|----------------------------------------|-----------------------------|
| RAG Evaluation     | Validates grounding gate pass rates    | `npm run rag-eval`          |
| MCP Smoke Test     | Validates tool contracts + security    | `npm run mcp-smoke-test`    |
| Lifecycle QA       | Verifies RLS + purge idempotency       | `psql -f qa/lifecycle/*.sql`|
| Roadmap QA         | Verifies dependency graph integrity    | `psql -f qa/roadmap/*.sql`  |

4. CI & RELEASE GATES
---------------------
- PR Checklist Reminder: A GitHub Action (.github/workflows/pr-checklist-reminder.yml)
  posts a nudge comment if the checklist is missing from a sensitive PR.
- Grounding SLO Gate: A mandatory runtime gate in `ai-task-router` that prevents
  low-quality AI responses from reaching users. Default: min_score 0.80.

5. RELEASE RUNBOOK
------------------
- For new features (Rollups, Retention, MCP):
  1. Verify migrations apply cleanly on dev.
  2. Run the specific QA harness from Section 3.
  3. Perform a manual security review (see docs/mcp/SECURITY_REVIEW.md).
  4. Enable the feature flag (e.g., `roadmap_enabled` on `packs` table).
  5. Monitor `lifecycle_audit_events` for errors during first few runs.
