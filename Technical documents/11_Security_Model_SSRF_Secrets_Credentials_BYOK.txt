1. SECURITY DESIGN PRINCIPLES
-------------------------------
  a) Defense-in-depth: Multiple independent security layers
  b) RLS-first: All data access decisions enforced at DB level
  c) Service role segregation: Only Edge Functions hold service_role key
  d) Zero secrets in transit: Credentials stored in Vault, never returned to browser
  e) Fail-closed: Ambiguous authorization → deny
  f) Bounded inputs: All user-supplied strings capped before processing

2. SSRF PREVENTION (parseAndValidateExternalUrl)
--------------------------------------------------
  Location: supabase/functions/_shared/external-url-policy.ts
  
  Called by: EVERY Edge Function before any fetch() to an external URL.
  Rule: Any URL derived from user input, connector config, or pack_sources
        MUST be validated before fetch().

  Default Policy (URLPolicy):
    allowHttps: true
    allowHttp: false              (HTTP rejected by default)
    allowAnyHost: false
    disallowPrivateIPs: true
    allowedPorts: [80, 443]

  Blocking Rules (applied in order):
    1. Non-http(s) protocol → reject
    2. Embedded credentials (user:pass@host) → reject
    3. Non-allowlisted port → reject
    4. Raw IP literal (IPv4 or IPv6) → reject (unless allowAnyHost=true AND still blocked if private)
    5. Non-allowlisted hostname → reject (if not allowAnyHost)
    6. localhost/private IP patterns → reject (regex: /localhost/, /^127\./, /^10\./, etc.)

  Connector-specific allowlists:
    Each connector provides { allowedHostSuffixes: ['.atlassian.net', ...] }

  What's blocked:
    - localhost, 127.0.0.1
    - 169.254.169.254 (AWS/GCP metadata)
    - 10.x.x.x, 172.16-31.x.x, 192.168.x.x (RFC1918 private)
    - 0.0.0.0
    - All link-local addresses
    - Non-standard ports (anything not 80 or 443 by default)

  Note: IP literal blocking is best-effort for CIDR ranges (full CIDR validation
  would require a library). Hostname-to-IP resolution (DNS rebinding) is NOT
  currently mitigated by the policy module. See 19_Known_Gaps.txt.

3. SECRET DETECTION / REDACTION PIPELINE
-----------------------------------------
  Location: supabase/functions/_shared/secret-patterns.ts (SINGLE SOURCE OF TRUTH)

  Pattern List (SECRET_PATTERNS array):
    Name                    | Severity | Pattern match
    ------------------------|----------|--------------------------------
    aws_access_key          | critical | AKIA[0-9A-Z]{16}
    aws_session_token       | critical | FwoGZXIvYXdzE.... (STS prefix)
    github_pat              | critical | ghp_[A-Za-z0-9]{36+}
    github_oauth            | critical | gho_[A-Za-z0-9]{36+}
    github_app_token        | critical | ghs_[A-Za-z0-9]{36+}
    github_fine_grained     | critical | github_pat_...
    openai_api_key          | critical | sk-...T3BlbkFJ...
    openai_project_key      | critical | sk-proj-...
    anthropic_api_key       | critical | sk-ant-...
    slack_token             | critical | xox[bpoas]-...
    stripe_key              | critical | [sr]k_(live|test)_...
    sendgrid_key            | critical | SG....
    jwt_token               | high     | eyJ...eyJ...
    bearer_token_assignment | high     | bearer/authorization context
    database_url            | critical | postgres://... mysql://... etc.
    private_key_block       | critical | -----BEGIN PRIVATE KEY-----
    google_api_key          | high     | AIza[35 chars]
    google_oauth_secret     | critical | GOCSPX-...
    supabase_service_role   | critical | JWT matching service key format
    slack_webhook           | high     | hooks.slack.com/services/...
    twilio_key              | high     | SK[32 hex chars]
    aws_secret_key          | medium   | 40-char base64 (context-dependent)
    generic_secret_assign.  | medium   | password=..., api_key=..., etc.

  Detection: detectSecrets(text) → SecretMatch[] (no secret values logged)
  Redaction: redactText(text) → { redactedText, secretsFound, matchedPatterns, redactionRatio }
  
  Decision logic (assessChunkRedaction):
    - redactionRatio > 0.5 → action: 'exclude', is_redacted=true
    - hasCritical AND ratio <=0.5 → action: 'redact_and_index', is_redacted=false
    - no critical, ratio <=0.3 → action: 'redact_and_index', is_redacted=false
    - no secrets → action: 'clean', is_redacted=false

  Two-pass defense:
    Pass 1 (Ingestion): assessChunkRedaction() on raw content before storing
    Pass 2 (Retrieval): redactSpans() in ai-task-router before evidence spans
                        are included in the AI prompt

4. CREDENTIALS AND VAULT STORAGE
----------------------------------
  Location: supabase/functions/_shared/credentials.ts
  Location: supabase/migrations/20260319000000_credential_vault_storage.sql

  All API tokens, OAuth tokens, and secrets stored in:
    Supabase Vault (supabase_vault.secrets) via RPC
    Never in: pack_sources.config (that column is for non-secret settings)

  Credential retrieval:
    getSourceCredential(supabase, source_id, key_name) → string | null
    Uses RPC: vault.read_secret(vault_id) → decrypted value
    Caller (Edge Function) has service_role key; secret never sent to browser

  Credential storage:
    storeSourceCredential(supabase, source_id, key_name, secret_value)
    Uses RPC: vault.create_secret(value, name) → vault_id
    Vault ID stored in pack_sources.metadata.vault_ids[key_name]

  Google OAuth tokens:
    Stored in google_oauth_tokens table (encrypted via Vault reference)
    managed by google-oauth-callback edge function

  NEVER stored in:
    - pack_sources.config
    - knowledge_chunks.metadata
    - Any PostgREST-readable column
    - Client-side localStorage

5. ROLE-BASED ACCESS CONTROL (RBAC)
-------------------------------------
  Roles (pack_members.role):
    'learner' — can view pack content, take quizzes, view own progress
    'author'  — can manage sources, ingestion, lifecycle, roadmaps, view Trust Console
    'admin'   — same as author (currently no distinct admin privileges beyond author)

  Key RBAC helpers:
    has_pack_access(user_id, pack_id, required_role) → boolean
      - Used in RLS policies for: lifecycle, audit, roadmap management
    is_pack_member(user_id, pack_id) → boolean
      - Used for content read access (playlists, modules)

  RLS enforcement means RBAC is enforced at the DB level, not just in code.
  Edge Functions that bypass RLS (using service_role) do their own RBAC checks.

TABLE: pack_docs
  SELECT: allowed if (status='published' AND is_pack_member()) OR has_pack_access(author).
  INSERT/UPDATE: allowed if has_pack_access(author).

6. EDGE FUNCTION SECURITY
--------------------------
  ai-task-router:
    - JWT validated via supabase.auth.getUser()
    - Rate limited (in-memory, 30 req/min per user)
    - Inputs sanitized/capped
    - Spans second-pass redacted before AI prompt
    - No raw credential access

  retrieve-spans:
    - JWT validated
    - Pack membership verified via pack_members table
    - CORS: allowlist-based (ALLOWED_ORIGINS env var)
    - Query capped at 500 chars
    - Returns 403 for non-members

  purge-source:
    - JWT validated
    - Role check: pack_members.role must be 'author' or 'admin'
    - dry_run mode available for safety
    - All deletes logged to lifecycle_audit_events

  github-webhook:
    - HMAC SHA-256 signature verified (X-Hub-Signature-256)
    - Repository cross-check before acting

7. MCP SECURITY
----------------
  - Tool Registry: policy flags ('read-only' vs 'mutating') enforced by router.
  - Output Caps: Strict token limits (50k chars) and secret assessment on every tool output.
  - Auth: Mandatory JWT + pack-access verification.

8. NO-SECRET-LOGGING POLICY
-----------------------------
  Rules enforced in code:
    - detectSecrets returns SecretMatch metadata, NOT the secret value
    - Redaction warnings log: span_id and path — NOT the matched text
    - No env var values are logged anywhere
    - Error messages returned to clients are generic (not stack traces)

9. SECURITY INVARIANTS (NON-NEGOTIABLE)
-----------------------------------------
  1. SSRF: Every URL from user/connector config MUST be validated before fetch()
  2. Secrets: assessChunkRedaction() on every chunk at ingestion time (primary gate)
  3. Credentials: NEVER in pack_sources.config; always in Vault
  4. Service role: Only held by Edge Functions, never returned to browser
  5. RLS: Enabled on ALL tables containing multi-tenant data
  6. No raw IP literals: IP literals rejected by external-url-policy.ts by default
  7. No PUBLIC execute on sensitive RPCs (revoke PUBLIC; grant service_role only)
  8. ai_audit_events: No UPDATE/DELETE policy (append-only)

  8. ai_audit_events: No UPDATE/DELETE policy (append-only)

10. KNOWN LIMITATIONS (also see 19_Known_Gaps)
---------------------------------------------------
  - DNS rebinding: The SSRF policy validates the URL string but cannot detect
    if a resolved IP is private (no DNS resolution at validation time).
    Mitigation note in 19_Known_Gaps_Tech_Debt_and_Roadmap.txt.
  - Rate limiting is in-memory (Deno isolate); resets on cold start.
    For distributed rate limiting, a Redis-backed approach is needed.
  - AWS secret key pattern (medium severity) is broad (40-char base64)
    and may produce false positives. Gated by context in pattern description.
