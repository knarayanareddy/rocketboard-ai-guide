1. PURPOSE
-----------
Lifecycle Controls satisfy enterprise compliance requirements:
  - Remove data no longer needed (per-source purge)
  - Automatically expire old data (retention policies)
  - Audit every deletion action (lifecycle audit events)
  - Support legal hold (no-delete flag)

Lifecycle controls are ADDITIVE (do not change product logic).
All AI functionality is independent of rollups and lifecycle state.

2. PER-SOURCE PURGE
--------------------
  Edge Function: purge-source (supabase/functions/purge-source/index.ts)
  Endpoint: POST /functions/v1/purge-source
  Auth: JWT Bearer (must be pack author/admin)

  Request:
  { "pack_id": "uuid", "source_id": "uuid", "dry_run": false }

  dry_run=true response:
  { "dry_run": true, "would_delete": { "knowledge_chunks": N, "ingestion_jobs": N } }

  dry_run=false response:
  { "success": true, "deleted": { "knowledge_chunks": N, "ingestion_jobs": N },
    "audit_event_id": "uuid" }

  RBAC check:
    pack_members.role = 'author' or 'admin' required (server-side RBAC check)
    Returns 403 if not authorized.

  Underlying SQL (purge_source_v1 RPC):
    DELETE FROM knowledge_chunks WHERE pack_id=? AND source_id=?   → v_chunks_deleted
    DELETE FROM ingestion_jobs WHERE pack_id=? AND source_id=?     → v_jobs_deleted
    UPDATE pack_sources SET last_synced_at=NULL WHERE id=? AND pack_id=?
    Returns JSONB {knowledge_chunks: N, ingestion_jobs: N}

  Audit log written:
    lifecycle_audit_events INSERT {
      pack_id, actor_user_id, action: 'purge_source',
      target_type: 'source', target_id: source_id,
      parameters: {dry_run: bool},
      rows_deleted: {knowledge_chunks: N, ingestion_jobs: N},
      status: 'completed'
    }

  Idempotency: Purging an already-purged source returns 0 rows deleted, still writes audit log.

3. RETENTION POLICIES (pack_lifecycle_policies)
-------------------------------------------------
  Table: pack_lifecycle_policies
  Primary key: pack_id (one policy per pack)
  
  Columns:
    retention_rag_metrics_days INT DEFAULT 90
    retention_ingestion_jobs_days INT DEFAULT 90
    retention_audit_days INT DEFAULT 365 (added in ai_audit migration)
    legal_hold BOOLEAN DEFAULT false
  
  RLS: Pack authors/admins can read and write their pack's policy.
  Trigger: trg_pack_lifecycle_policies_updated_at (sets updated_at on change)

4. AUTOMATED RETENTION CLEANUP (lifecycle-retention-job)
----------------------------------------------------------
  Edge Function: lifecycle-retention-job
  Endpoint: POST /functions/v1/lifecycle-retention-job
  Auth: CRON_AUTH_TOKEN bearer header (not user JWT)
  Trigger: Scheduled cron (must be configured externally, e.g., pg_cron or Vercel cron)
  
  Algorithm (per pack):
    1. Fetch all packs + their pack_lifecycle_policies
    2. For packs without a policy row: use defaults (90/90/365 days)
    3. If legal_hold=true: SKIP (do not delete anything)
    4. Delete rag_metrics WHERE pack_id=? AND created_at < (now() - N days)
       (in batches of 1000 rows to avoid timeouts)
    5. Delete ingestion_jobs WHERE pack_id=? AND started_at < (now() - N days)
       (in batches of 1000 rows)
    6. Delete ai_audit_events WHERE pack_id=? AND created_at < (now() - N_audit days)
       (in batches of 1000 rows)
    7. INSERT lifecycle_audit_events {
         action: 'retention_cleanup',
         target_type: 'table',
         rows_deleted: { rag_metrics: N, ingestion_jobs: N, ai_audit_events: N },
         status: 'completed'
       }
  
  Response: { success: true, processed_packs: N, errors: [...] }
  
  Notes:
    - Failures for individual packs are caught and logged; do not abort entire run
    - knowledge_chunks are NOT deleted by retention (only by explicit purge)

5. LIFECYCLE AUDIT EVENTS (lifecycle_audit_events)
----------------------------------------------------
  table: lifecycle_audit_events
  Purpose: Immutable record of every purge and retention action
  
  Columns:
    pack_id UUID
    actor_user_id UUID (null for cron jobs)
    action TEXT ('purge_source' | 'retention_cleanup')
    target_type TEXT ('source' | 'table')
    target_id UUID (source_id for purge; null for cleanup)
    parameters JSONB (e.g., {dry_run: true})
    rows_deleted JSONB (e.g., {knowledge_chunks: 150, ingestion_jobs: 12})
    status TEXT ('started' | 'completed' | 'failed')
    error_message TEXT (if failed)
    created_at TIMESTAMPTZ
  
  RLS: Pack authors/admins can SELECT only (no INSERT from client; only service role)
  No UPDATE or DELETE policies.

6. AI AUDIT EVENTS CLEANUP
----------------------------
  The lifecycle-retention-job also purges ai_audit_events older than
  retention_audit_days (default: 365 days), UNLESS legal_hold=true.
  
  Deleted count reported in lifecycle_audit_events.rows_deleted.ai_audit_events.
  
  Note: ai_audit_events themselves are the AI governance audit trail.
        Their cleanup is intentionally conservative (1 year default).

7. LIFECYCLE UI (LifecycleSettings.tsx)
-----------------------------------------
  Route: /packs/:packId/settings/lifecycle
  Access: Author/admin only
  
  Features:
    - View/edit retention policy (days for rag_metrics, ingestion_jobs)
    - View legal_hold status (toggle requires admin confirmation)
    - View per-source list (with chunk counts)
    - Dry-run purge a source → see what would be deleted
    - Execute purge a source → confirmation dialog required
    - View recent audit events (last 10 rows)

8. LIFECYCLE HEALTH PANEL (LifecycleHealthPanel.tsx)

  Location: TrustDashboard.tsx (embedded panel)
  Shows:
    - Current retention policy summary
    - legal_hold indicator
    - Recent audit events (last 3)
    - Link: "Manage Lifecycle Settings" → /packs/:packId/settings/lifecycle

9. REPEATABLE QA SCRIPTS (supabase/qa/lifecycle/)
---------------------------------------------------
  Folder: supabase/qa/lifecycle/
  README.md:
    - Prerequisites: 2 test users (author + learner), test pack
    - How to plug in UUIDs (replace :pack_id, :author_user_id, :learner_user_id)
    - Run order: 00 → 01 → 02 → 03 → 04 → 05
    - Expected outcomes for each script
    - Cleanup instructions

  Script files:
    00_vars.sql (or 00_vars.template.sql)
      — Variable placeholders: :pack_id, :author_user_id, :source_a_id, :source_b_id
      — Set once, referenced by all other scripts

    01_seed.sql
      — Insert test chunks for source_a and source_b
      — Insert test rag_metrics for the pack

    02_assertions_learner.sql
      — Run as learner user (set role check)
      — Assertions:
          lifecycle_audit_events SELECT → 0 rows (access denied)
          purge_source_v1() call → error (permission denied)

    03_assertions_author.sql
      — Run as author user
      — Assertions:
          lifecycle_audit_events SELECT → succeeds
          pack_lifecycle_policies SELECT → succeeds
          purge_source_v1(source_a) → returns row counts

    04_purge_checks.md
      — Manual checklist:
          Dry run purge source A → verify counts match SELECT COUNT(*)
          Execute purge source A → verify chunks=0 for A, B intact
          Execute again (idempotent) → 0 chunks deleted, audit event written

    05_retention_checks.md
      — Manual checklist:
          Set retention_rag_metrics_days=1 for test pack
          Insert old rag_metrics (created_at = 3 days ago)
          Call lifecycle-retention-job
          Verify old metrics deleted; recent metrics intact
          Verify lifecycle_audit_events written

10. KEY INVARIANTS
-------------------
  - Purge is ALWAYS scoped to (pack_id, source_id). No pack-wide deletes.
  - legal_hold=true prevents ALL retention cleanup for any table.
  - Vault credentials are NEVER purged by purge-source or lifecycle-retention-job.
  - Every purge and cleanup action writes a lifecycle_audit_events row.
  - ai_audit_events is append-only from the application (no delete policy for users).
    Only lifecycle-retention-job can delete them (service role).

11. FAILURE MODES
------------------
  Issue: purge-source returns 403
  Cause: User does not have author/admin role for the pack
  Fix: Verify pack_members.role for the calling user

  Issue: lifecycle-retention-job returns auth error
  Cause: CRON_AUTH_TOKEN not set or mismatched
  Fix: Verify CRON_AUTH_TOKEN env var matches the value used in cron trigger

  Issue: Purge idempotency: second purge fails with error
  Cause: purge_source_v1 should return 0 rows, not error
  Fix: purge_source_v1 uses DELETE with WHERE → 0 rows affected is not an error

  Issue: legal_hold active but needed to run cleanup
  Cause: legal_hold=true was set
  Fix: Authorized admin must explicitly set legal_hold=false before cleanup can run
